# Домашнє завдання до Теми Структурні патерни: Декоратор та Замісник

### **Опис завдання**

Мета домашнього завдання — навчитися поєднувати два структурні патерни у спрощеній моделі сервісу повідомлень:

- **Декоратор (Decorator)** — дозволяє розширювати функціональність сервісу повідомлень без зміни його коду;
- **Замісник (Proxy)** — обмежує частоту звернень до сервісу, імітуючи rate-limit поведінку, як у реальних API.

Ваше завдання — реалізувати консольну утиліту, яка:

- надає інтерфейс `send(message: string): void`;
- надсилає повідомлення через базовий сервіс в термінал;
- обгортає цей сервіс у декоратори, які змінюють текст повідомлення;
- застосовує Замісник, що блокує надсилання повідомлень, якщо вони надходять надто часто;
- демонструє розмежування відповідальностей, відкритість до розширення і чітке дотримання структурних патернів.

## Завдання

### Структура проєкту

```sh
/
├── src/
│   ├── IMessageService.ts
│   ├── MessageService.ts
│   ├── decorators.ts
│   ├── RateLimitProxy.ts
│   └── main.ts
├── tsconfig.json
├── package.json
└── README.md
```


У файлі IMessageService.ts опишіть інтерфейс IMessageService, який містить метод:

```typescript
export interface IMessageService {
  send(message: string): void;
}
```

У файлі decorators.ts реалізуйте два декоратори методу згідно синтаксису декораторів TypeScript:
   
```typescript
@withTimestamp — додає на початок повідомлення часову мітку [YYYY-MM-DD HH:mm:ss];
@uppercase — перетворює повідомлення повністю на великі літери.
```

У файлі MessageService.ts створіть клас MessageService, який реалізує IMessageService та має метод send, до якого застосовано декоратори @withTimestamp і @uppercase. Порядок декораторів має вплив: спочатку час, потім регістр.


У файлі RateLimitProxy.ts реалізуйте клас RateLimitProxy, який реалізує IMessageService, приймає wrappee: IMessageService та intervalMs: number. Якщо метод send викликається раніше, ніж через intervalMs після попереднього виклику, повідомлення не надсилається, а виводиться [RateLimit] skipped.


У файлі main.ts реалізуйте CLI-додаток, який демонструє використання створених патернів. Наприклад:

```typescript
import { MessageService } from './MessageService';
import { createRateLimitProxy } from './RateLimitProxy';

const messageService = new MessageService();
// Обмежуємо відправку до 1 повідомлення в секунду
const service = createRateLimitProxy(messageService, 1000);

console.log("Тестуємо систему анти-спаму:");
service.send("Привіт! Як справи?");
// Спроба спаму - буде заблоковано
service.send("Чому не відповідаєш?");

// Чекаємо секунду і надсилаємо знову
setTimeout(() => {
    service.send("Це повідомлення вже пройде, бо ми почекали 1 секунду");
}, 1100);
```

### Поради

Ми працюємо з Декоратором та Замісником (Proxy). Обидва патерни створюють «обгортку» навколо об'єкта, але з абсолютно різною метою.
Ось мої поради, як майстерно виконати це завдання:

#### 1. Декоратор (Decorator): Трансформація даних

Декоратори в TypeScript (через символ @) дозволяють "підмінити" або розширити логіку методу ще на етапі визначення класу.

На що звернути увагу: Згідно з оновленим стандартом TypeScript, декоратор методу приймає два аргументи: власне метод (target) та контекст. Ваше завдання - повернути нову функцію, яка спочатку модифікує вхідний рядок (додає дату або змінює регістр), а потім викликає оригінальний метод.
Важливо про порядок: У завданні вказано, що порядок має значення. Декоратори виконуються зверху вниз (або знизу вгору залежно від того, як ви їх обгорнете). Переконайтеся, що часова мітка додається до того, як текст стане великими літерами, або навпаки, згідно з очікуваним результатом: [DATE] ПРИВІТ.

#### 2. Замісник (Proxy): Контроль доступу

Якщо Декоратор змінює зміст повідомлення, то Proxy контролює, чи взагалі воно буде надіслане.

На що звернути увагу: Клас RateLimitProxy повинен зберігати час останнього успішного виклику (lastSentTime).
Логіка роботи: При кожному виклику send() проксі порівнює поточний час із lastSentTime.
Якщо різниця більша за intervalMs - оновлюємо час і передаємо виклик реальному сервісу.
Якщо менша - блокуємо виклик і виводимо [RateLimit] skipped.

Порада: Proxy - це ідеальний спосіб реалізувати логіку, яка не стосується основного бізнесу (як-от безпека, логування або, як тут, ліміти), не захаращуючи основний клас MessageService.

#### 3. Налаштування TypeScript

Оскільки ви використовуєте декоратори, звичайних налаштувань tsconfig.json може бути замало.

Порада: Переконайтеся, що у вашому tsconfig.json увімкнено опцію "experimentalDecorators": true. Без цього TypeScript видасть помилку при спробі використати символ @.


#### 4. Робота через інтерфейс IMessageService
Це "золоте правило" SOLID, про яке ми говорили раніше.

На що звернути увагу: Ваш main.ts не повинен знати, працює він з реальним сервісом чи з проксі. Обидва вони реалізують IMessageService. Це дозволяє вам легко додавати нові шари обгорток у майбутньому (наприклад, CachingProxy або AuthProxy), не змінюючи код, який надсилає повідомлення.


#### Чек-лист для самоперевірки:

- Порядок декораторів: Чи правильно відображається дата та регістр? (Чи не "з'їдає" @uppercase символи в даті, якщо вони мають бути маленькими?)
- Точність Rate Limit: Чи справді проксі блокує повідомлення, надіслані через 100 мс, якщо ліміт - 1000 мс?
- Глибока інкапсуляція: Чи є поля lastSentTime у проксі та intervalMs приватними?
- Синтаксис: Чи використовуєте ви createRateLimitProxy як фабричну функцію, чи створюєте екземпляр через new? Завдання пропонує варіант із функцією - це зручний паттерн для створення проксі-об'єктів.

   Порада від мене: Це завдання демонструє принцип Open/Closed: ми розширили можливості сервісу (додали дату, регістр та ліміти), але при цьому не змінили жодного рядка коду всередині самого методу send базового класу. Саме так будуються плагіни та великі фреймворки (як-от NestJS або Angular)
